
## Назначение

Спецификация описывает полный цикл работы агента над новой фичей — от идеи до коммита в git-flow ветку. Процесс работает внутри ralph-loop, который активен от EXPLORE до ARCHIVE. Поддерживается feedback loop между разработкой и дизайном.

---

## Архитектура потока

```
ralph-loop active ──────────────────────────────────────┐
                                                         │
                                         ┌──────────┐    │
        ┌──────────┐  ┌──────────┐  ┌───▶│  VERIFY  │    │
   ────▶│ EXPLORE  │─▶│ PROPOSE  │─▶│    └────┬─────┘    │
        └──────────┘  └──────────┘  │    ┌────▼─────┐    │
                ▲            ▲      │    │  ARCHIVE  │────┘
                │            │      │    └──────────┘
                │   ┌────────┘      │         │
                │   │  design issue │    cancel-ralph
                │   │  found        │         │
                │   │  (feedback)   │       DONE
                │   └───────────────┘
                │
           ┌────┴────┐
           │  APPLY  │
           │  (TDD)  │
           └─────────┘
```

## Фазы процесса

### EXPLORE — Обсуждение

Вход: пользователь описывает фичу. Агент запускает `/ralph-loop`.

Шаги:
1. Загрузить скил **openspec-explore** — свободное обсуждение требований
2. Загрузить скил **brainstorming** — генерация идей, варианты подходов
3. Загрузить скил **test-driven-development** → определить тестовую стратегию (что и как тестировать)
4. Сформулировать DoD-критерии — это все задачи в `tasks.md`, которые будут созданы
5. Если по ходу выясняются новые детали — итерировать, оставаясь в EXPLORE

Результат: прояснённые требования, понимание тестовой стратегии, готовность к созданию change.

---

### PROPOSE — Пропозал

Вход: после EXPLORE. Агент загружает скил **openspec-propose** и создаёт OpenSpec change.

Шаги:
1. **Проверка текущей ветки**:
   - Если уже на `feature/*`, `release/*` или `hotfix/*` → остановиться, уточнить у пользователя: использовать текущую ветку или создать новую
   - Если на `develop` или `main` → продолжить
2. **Определение типа ветки** на основе описания фичи:
   - `feature/<kebab-name>` (по умолчанию) — новая функциональность
   - `release-*` — подготовка релиза
   - `hotfix/<kebab-name>` — критический багфикс
   - Агент предполагает тип и уточняет у пользователя
3. **Создание git ветки**:
   - `git checkout -b <type>/<name> <base>`
   - default base: `develop` (feature/release) / `master` (hotfix); если develop не существует → `main`
4. Создать change через `openspec new change "<name>"`
5. Создать артефакты: `proposal.md`, `design.md`, `tasks.md` в порядке зависимостей
6. DoD = все задачи в `tasks.md` выполнены

Feedback loop: если на этапе APPLY обнаружена проблема в дизайне → вернуться в PROPOSE, обновить артефакты.

---

### APPLY — Разработка

Вход: после PROPOSE. Агент загружает скил **openspec-apply-change** + **test-driven-development**.

Шаги:
1. Прочитать контекст (proposal, design, tasks)
2. Для каждой pending задачи:
   - **RED**: написать тест, убедиться что падает
   - **GREEN**: написать минимальный код, тест проходит
   - **REFACTOR**: улучшить код, тесты зелёные
3. Если в процессе вскрылась проблема в дизайне/задачах:
   - Поставить текущую задачу на паузу
   - Feedback loop → вернуться в PROPOSE, обновить артефакты
   - После обновления продолжить APPLY

Результат: все задачи выполнены, тесты проходят.

---

### VERIFY — Проверка

Вход: после APPLY (все задачи отмечены `[x]`).

Шаги:
1. `npm run test` — все тесты проходят
2. `npm run build` — сборка чистая
3. Если что-то упало → вернуться в APPLY и исправить

Результат: код готов к архивации.

---

### ARCHIVE — Архивация

Вход: после VERIFY. Агент загружает скил **openspec-archive-change**.

Шаги:
1. Выполнить архивацию change: `mv openspec/changes/<name> openspec/changes/archive/YYYY-MM-DD-<name>`
2. **Коммит** в git (с подтверждением пользователя):
   - Сообщение: `type(scope): краткое описание` (conventional commit)
   - Коммитятся только файлы, относящиеся к фиче
3. **Автоматически** (без запроса): `/cancel-ralph`

Результат: фича закоммичена в feature-ветку, ralph-loop завершён.

---

## Триггеры feedback loop

| Ситуация | Действие |
|----------|----------|
| В APPLY обнаружена проблема в дизайне | Поставить задачу на паузу → PROPOSE (обновить design.md/tasks.md) → продолжить APPLY |
| В EXPLORE выяснились новые требования | Итерировать в EXPLORE, пока требования не стабильны → PROPOSE |
| VERIFY упал | Вернуться в APPLY, исправить, повторить VERIFY |

---

## Используемые скилы

| Фаза | Скил |
|------|------|
| EXPLORE | openspec-explore, brainstorming, test-driven-development (стратегия) |
| PROPOSE | openspec-propose, git-flow-branch-creator |
| APPLY | openspec-apply-change, test-driven-development (full TDD) |
| VERIFY | — |
| ARCHIVE | openspec-archive-change |
